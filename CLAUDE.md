## Project Overview

An Intel SGX secure data lake prototype for confidential multi-hospital patient data analysis. Multiple hospitals encrypt their patient records and submit them to an SGX enclave, which decrypts and runs aggregate queries (AVG/MIN/MAX/COUNT) without exposing individual records.

## Build & Run

Before building, source the SGX SDK environment:
```bash
source /opt/intel/sgxsdk/environment
```

```bash
make            # Build everything
make clean      # Remove all build artifacts
./app           # Run the application
```

The build runs in simulation mode (`SGX_MODE=SIM`) — no SGX hardware required.


## Architecture

The codebase follows the SGX split-trust model:

**Untrusted side (`App/`)**: Modular host application. `app_main.cpp` runs the menu loop. CSV loading, encryption, attestation, upload, and query logic are each in separate modules.

**Trusted side (`Enclave/Enclave.cpp`)**: The enclave. Holds decrypted patient records and session keys in protected memory. Exposes ECALLs for attestation, key exchange, data upload, and query execution. Uses SGX tcrypto (`sgx_rijndael128GCM_decrypt`, `sgx_ecdsa_sign`) — never OpenSSL.

**Interface (`Enclave/Enclave.edl`)**: Defines the ECALL/OCALL boundary. After editing this file, the build regenerates `Enclave_t.{c,h}` (trusted wrappers) and `Enclave_u.{c,h}` (untrusted wrappers) automatically via `sgx_edger8r`.

**Shared types (`Include/types.h`)**: `PatientRecord`, `DCAPQuote`, `AttestationReport`, and related constants shared across the trust boundary.

## Key Data Structures (`Include/types.h`)

- `PatientRecord`: `{patient_id, age, temperature, blood_sugar, diagnosis}`
- `DCAPQuote`: attestation quote with MRENCLAVE, MRSIGNER, nonce, and ECDSA signature

## Attestation Flow

1. App generates a 16-byte random nonce
2. `ecall_generate_report()` — enclave embeds nonce in quote user_data and signs with ECDSA
3. App verifies: nonce match, MRENCLAVE value, MRSIGNER, and QE signature
4. `ecall_finish_key_exchange()` — enclave generates a random 16-byte AES session key and returns it
5. App encrypts patient records with that session key before uploading

## Current Limitations

- MRENCLAVE is hardcoded (not computed from actual enclave binary)
- Session key is generated by the enclave unilaterally (not proper ECDH)
- Data is in-memory only — lost on enclave restart
- Single-process only (no client/server separation)
